# Requirements Document

## Introduction

Helm V2 is a ground-up rebuild of the Helm desktop automation agent. The current system has degraded through iterative changes, resulting in broken web interaction (DevTools/DOM replaced by unreliable vision-based clicking), Paint tool misuse, non-replayable training recordings, excessive LLM calls, and no real-time video analysis capability. The rebuild delivers two core systems: a Desktop Automation Agent that reliably controls any Windows 11 application or website, and a Real-Time Video Analysis system for continuous monitoring, alerting, and commentary on streaming video content.

## Glossary

- **Helm**: The desktop automation agent system, consisting of a FastAPI web server (port 8765), an agent executor, and supporting modules
- **Clawmetheus**: An HTTP server (port 7331) providing low-level screen capture, mouse control, and keyboard input via Win32 APIs
- **Ollama**: A local model inference server (port 11434) hosting Qwen vision and language models on the local GPU
- **Kiro_Proxy**: An HTTP proxy server (port 8000) providing access to Claude cloud models (Opus, Haiku)
- **Vision_Module**: The component responsible for screen understanding using Qwen3.5:0.8B via Ollama, producing scene descriptions and state assessments
- **DOM_Inspector**: The component that interacts with browser DevTools to extract CSS selectors, element coordinates, and page structure for precise web element interaction
- **UIA_Module**: The component that uses Windows UI Automation APIs to discover and interact with desktop application UI elements
- **Knowledge_Base**: A collection of per-application JSON files containing startup sequences, keyboard shortcuts, tool locations, and learned behaviors
- **Model_Router**: The component that selects the appropriate LLM tier (Local Qwen, Haiku, or Opus) based on task complexity
- **Action_Registry**: The fixed library of proven action primitives (click, type, drag, etc.) that the agent composes into task plans
- **Frame_Capturer**: The component that continuously captures screenshots at 1-2 FPS from a specified screen region or browser tab
- **Event_Detector**: The component that compares frame descriptions across time to identify changes, objects, or conditions of interest
- **Alert_System**: The component that triggers notifications or actions when the Event_Detector identifies a matching condition
- **Training_Recorder**: The component that captures user demonstrations and converts them into replayable action sequences
- **Startup_Sequence**: A deterministic, ordered list of actions stored in the Knowledge_Base that launches and prepares a specific application
- **Action_Plan**: A batch of actions generated by the LLM planner, executed sequentially, then verified by vision

## Requirements

### Requirement 1: Web Element Interaction via DOM

**User Story:** As a user, I want Helm to click web elements using DevTools/DOM inspection instead of vision-based coordinate guessing, so that web interactions are precise and reliable.

#### Acceptance Criteria

1. WHEN a web page is loaded in the browser, THE DOM_Inspector SHALL extract element coordinates using CSS selectors from the browser DevTools protocol
2. WHEN the user requests interaction with a web element, THE Helm SHALL use DOM-derived coordinates for mouse clicks instead of vision-estimated coordinates
3. WHEN a CSS selector matches multiple elements, THE DOM_Inspector SHALL return all matching elements with their bounding rectangles and visible text
4. IF the DOM_Inspector fails to connect to the browser DevTools, THEN THE Helm SHALL log the connection error and fall back to UIA-based element discovery
5. WHEN a web element is clicked using DOM coordinates, THE Vision_Module SHALL verify the click result by comparing before and after screenshots

### Requirement 2: Vision for Screen Understanding

**User Story:** As a user, I want Helm to use the local vision model for understanding what is on screen rather than for determining click coordinates, so that vision is used where it excels and clicking is precise.

#### Acceptance Criteria

1. THE Vision_Module SHALL use Qwen3.5:0.8B via Ollama to generate scene descriptions from screenshots
2. WHEN a screenshot is captured, THE Vision_Module SHALL produce a structured description including active application, visible UI elements, and current state within 1 second
3. THE Vision_Module SHALL NOT be used to estimate pixel coordinates for clicking UI elements
4. WHEN an action completes, THE Vision_Module SHALL compare before and after screenshots to verify the action produced the expected result
5. IF the Vision_Module determines an action failed verification, THEN THE Model_Router SHALL escalate to a higher-tier LLM for replanning

### Requirement 3: Keyboard-First Interaction

**User Story:** As a user, I want Helm to prefer keyboard shortcuts over mouse clicks, so that interactions are faster and more reliable.

#### Acceptance Criteria

1. WHEN the Knowledge_Base contains a keyboard shortcut for a requested action, THE Helm SHALL use the keyboard shortcut instead of a mouse click
2. THE Knowledge_Base SHALL store keyboard shortcuts for each known application in the per-application JSON configuration
3. WHEN no keyboard shortcut is available for an action, THE Helm SHALL fall back to mouse-based interaction using DOM coordinates (web) or UIA elements (desktop)
4. WHEN navigating menus or dialogs, THE Helm SHALL use Tab, Arrow keys, and Enter before attempting mouse clicks

### Requirement 4: Desktop App Interaction via UIA

**User Story:** As a user, I want Helm to use Windows UI Automation to interact with desktop application elements, so that clicks on native apps are precise without relying on vision coordinates.

#### Acceptance Criteria

1. WHEN interacting with a desktop application, THE UIA_Module SHALL discover UI elements by automation ID, name, or control type
2. WHEN a UI element is found via UIA, THE UIA_Module SHALL return the element bounding rectangle for precise click targeting
3. IF a UI element is not found via UIA within 5 seconds, THEN THE Helm SHALL fall back to vision-guided interaction with the Vision_Module providing scene context
4. WHEN the UIA_Module discovers elements, THE UIA_Module SHALL cache the element tree for the active window to avoid repeated tree walks

### Requirement 5: Tiered LLM Model Routing

**User Story:** As a user, I want Helm to route LLM calls to the appropriate model tier based on task complexity, so that simple tasks are fast and complex tasks get the reasoning they need.

#### Acceptance Criteria

1. THE Model_Router SHALL support three tiers: Local (Qwen via Ollama), Fast (Claude Haiku via Kiro_Proxy), and Smart (Claude Opus via Kiro_Proxy)
2. WHEN executing a pre-planned action step, THE Model_Router SHALL use the Local tier for action parameter generation
3. WHEN making routine decisions such as next-step selection, THE Model_Router SHALL use the Fast tier
4. WHEN performing complex planning, error recovery, or multi-step replanning, THE Model_Router SHALL use the Smart tier
5. IF the Local tier is unavailable or returns an unparseable response, THEN THE Model_Router SHALL automatically escalate to the Fast tier
6. IF the Fast tier is unavailable, THEN THE Model_Router SHALL escalate to the Smart tier

### Requirement 6: Batch Action Planning and Execution

**User Story:** As a user, I want Helm to plan batches of actions and execute them together instead of making an LLM call per action, so that task execution is faster.

#### Acceptance Criteria

1. WHEN a task is received, THE Helm SHALL generate an Action_Plan containing a batch of sequential actions before beginning execution
2. THE Helm SHALL execute each action in the Action_Plan sequentially without additional LLM calls between steps
3. WHEN all actions in a batch complete, THE Vision_Module SHALL verify the cumulative result against the expected outcome
4. IF batch verification fails, THEN THE Model_Router SHALL use the Smart tier to generate a corrective Action_Plan
5. WHEN executing in reactive mode (games, dynamic web content), THE Helm SHALL use single-action planning with the Fast tier instead of batch planning

### Requirement 7: Knowledge Base Driven Startup Sequences

**User Story:** As a user, I want Helm to use deterministic startup sequences from the knowledge base to launch and prepare applications, so that app initialization is reliable and repeatable.

#### Acceptance Criteria

1. THE Knowledge_Base SHALL store a Startup_Sequence for each known application as an ordered list of actions
2. WHEN a task requires an application that is not currently running, THE Helm SHALL execute the Startup_Sequence from the Knowledge_Base to launch and prepare the application
3. WHEN a Startup_Sequence completes, THE Vision_Module SHALL verify the application is in the expected ready state
4. IF the Startup_Sequence verification fails, THEN THE Helm SHALL retry the sequence once before escalating to the Smart tier for recovery
5. WHEN a user demonstrates a new application launch, THE Training_Recorder SHALL capture the demonstration and store it as a new Startup_Sequence in the Knowledge_Base

### Requirement 8: Training Recordings as Replayable Sequences

**User Story:** As a user, I want my training demonstrations to be converted into replayable action sequences, so that Helm can repeat what I showed it.

#### Acceptance Criteria

1. WHEN the user starts a training recording, THE Training_Recorder SHALL capture each user input event (mouse click, key press, drag) with timestamps and screen coordinates
2. WHEN the user stops a training recording, THE Training_Recorder SHALL convert the captured events into a sequence of Action_Registry primitives
3. THE Training_Recorder SHALL store each replayable sequence as a named skill in the Knowledge_Base
4. WHEN a stored skill is replayed, THE Helm SHALL execute the action sequence using the same Action_Registry primitives used during recording
5. WHEN replaying a skill, THE Vision_Module SHALL verify each action step completed as expected before proceeding to the next step
6. IF a replay step fails verification, THEN THE Helm SHALL pause replay and escalate to the Smart tier for adaptive recovery

### Requirement 9: Paint Tool Correct Usage

**User Story:** As a user, I want Helm to use Paint tools correctly (e.g., not use fill as pencil, not draw black on black), so that drawing tasks produce visible, correct results.

#### Acceptance Criteria

1. WHEN selecting a Paint tool, THE Helm SHALL verify the correct tool is active by checking the UIA element state before proceeding with drawing
2. WHEN drawing on the canvas, THE Helm SHALL verify the selected foreground color contrasts with the canvas background color
3. WHEN using the fill tool, THE Helm SHALL target a specific point within a closed region and verify the fill did not leak beyond the intended area
4. WHEN using shape tools (Ellipse, Rectangle, Line), THE Helm SHALL select the shape from the Shapes gallery dropdown before attempting to draw
5. THE Knowledge_Base paint configuration SHALL specify which tools require gallery expansion and which are directly accessible in the toolbar
6. WHEN a drawing action produces no visible change on the canvas, THE Vision_Module SHALL flag the action as failed and THE Helm SHALL check tool state and color selection before retrying


### Requirement 10: Real-Time Frame Capture

**User Story:** As a user, I want Helm to continuously capture frames from a screen region or browser tab at 1-2 FPS, so that video content can be analyzed in real time.

#### Acceptance Criteria

1. WHEN the user specifies a screen region (x, y, width, height), THE Frame_Capturer SHALL capture screenshots of that region at a configurable rate between 1 and 2 frames per second
2. WHEN the user specifies a browser tab, THE Frame_Capturer SHALL capture the visible content of that tab at the configured frame rate
3. THE Frame_Capturer SHALL deliver each captured frame to the Vision_Module for description within 100 milliseconds of capture
4. WHILE frame capture is active, THE Frame_Capturer SHALL maintain the configured frame rate without dropping below 0.5 FPS under normal system load
5. WHEN the user stops frame capture, THE Frame_Capturer SHALL release all associated resources within 2 seconds

### Requirement 11: Real-Time Video Description

**User Story:** As a user, I want Helm to generate text descriptions of each captured video frame using the local vision model, so that I get real-time understanding of video content.

#### Acceptance Criteria

1. WHEN a frame is received from the Frame_Capturer, THE Vision_Module SHALL generate a text description of the frame content using Qwen3.5:0.8B
2. THE Vision_Module SHALL produce each frame description within 1 second of receiving the frame
3. WHEN in commentary mode, THE Vision_Module SHALL stream frame descriptions to the web UI as a continuous text feed
4. THE Vision_Module SHALL include detected objects, people, actions, and scene context in each frame description
5. WHEN consecutive frames produce identical descriptions, THE Vision_Module SHALL suppress duplicate descriptions and report only changes

### Requirement 12: Event Detection Across Frames

**User Story:** As a user, I want Helm to detect specific events by comparing frame descriptions over time, so that I can be alerted when something important happens.

#### Acceptance Criteria

1. WHEN the user defines a watch condition (e.g., "coyote", "person", "package"), THE Event_Detector SHALL compare each new frame description against the condition
2. WHEN a frame description matches a watch condition, THE Event_Detector SHALL emit an event with the timestamp, frame, description, and matched condition
3. THE Event_Detector SHALL maintain a sliding window of the last 30 seconds of frame descriptions for temporal context
4. WHEN detecting change events, THE Event_Detector SHALL compare the current frame description against the previous frame description and flag meaningful differences
5. THE Event_Detector SHALL NOT emit duplicate events for the same ongoing condition within a configurable cooldown period (default 30 seconds)

### Requirement 13: Alert System

**User Story:** As a user, I want Helm to trigger notifications when specific video conditions are detected, so that I am alerted to important events without watching the screen.

#### Acceptance Criteria

1. WHEN the Event_Detector emits an event matching a user-configured alert rule, THE Alert_System SHALL trigger a desktop notification containing the event description
2. WHEN an alert is triggered, THE Alert_System SHALL play an audible notification sound
3. THE Alert_System SHALL log each triggered alert with timestamp, condition, frame description, and the captured frame image
4. WHEN multiple alerts fire within 10 seconds, THE Alert_System SHALL batch them into a single notification summarizing all triggered conditions
5. THE Alert_System SHALL expose alert history through the Helm web UI at port 8765

### Requirement 14: Hybrid Execution Modes

**User Story:** As a user, I want Helm to support both plan-then-execute mode for structured tasks and reactive mode for dynamic tasks, so that the agent adapts to different task types.

#### Acceptance Criteria

1. WHEN a task targets a structured application (e.g., Paint, file management), THE Helm SHALL use plan-then-execute mode: generate a full Action_Plan, execute the batch, then verify
2. WHEN a task targets a dynamic environment (e.g., games, live web content), THE Helm SHALL use reactive mode: observe screen state, decide one action, execute, observe again
3. THE Knowledge_Base SHALL specify the preferred execution mode for each known application
4. WHEN the user does not specify an execution mode, THE Helm SHALL default to plan-then-execute mode
5. WHILE in reactive mode, THE Model_Router SHALL use the Fast tier for each decision cycle to maintain responsiveness

### Requirement 15: VRAM and Resource Management

**User Story:** As a user, I want Helm to manage GPU VRAM efficiently so that both Qwen models (0.8B and 4B) can run simultaneously within the 8GB VRAM budget.

#### Acceptance Criteria

1. THE Helm SHALL configure Ollama to load both Qwen3.5:0.8B (~1GB) and Qwen3.5:4B (~3.4GB) simultaneously, staying within the 8GB VRAM limit
2. WHEN the Vision_Module requires fast frame analysis, THE Model_Router SHALL route to Qwen3.5:0.8B
3. WHEN the Vision_Module requires detailed scene analysis, THE Model_Router SHALL route to Qwen3.5:4B
4. IF VRAM usage exceeds 7.5GB, THEN THE Helm SHALL unload the 4B model and operate with only the 0.8B model until VRAM is available
5. THE Helm SHALL log GPU VRAM usage at startup and when model loading or unloading occurs

### Requirement 16: Web UI Integration

**User Story:** As a user, I want to control both the desktop automation agent and the video analysis system through the existing Helm web UI, so that I have a single interface for all Helm capabilities.

#### Acceptance Criteria

1. THE Helm web UI SHALL provide controls to start, stop, and configure video analysis sessions (region selection, frame rate, watch conditions)
2. THE Helm web UI SHALL display the real-time commentary stream from the Vision_Module during active video analysis
3. THE Helm web UI SHALL display alert history with timestamps, descriptions, and captured frame thumbnails
4. THE Helm web UI SHALL provide controls to start and stop training recordings
5. THE Helm web UI SHALL provide a skill library view showing all stored replayable sequences with the ability to trigger replay
6. WHEN a video analysis session is active, THE Helm web UI SHALL display the current frame rate and processing latency
